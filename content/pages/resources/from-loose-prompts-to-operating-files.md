# **Section 1: The Problem With Loose Prompts**

## **Eyebrow**

THE PROBLEM

## **Headline**

Private prompts can become hidden production infrastructure.

## **Body**

A prompt may start as a shortcut.

A caption prompt.  
A translation prompt.  
A product copy prompt.  
A clip suggestion prompt.  
A briefing prompt.  
A metadata prompt.  
A delivery checklist prompt.

At first, that feels harmless.

But once the prompt supports real media, it starts shaping the company’s production process.

The question becomes:

Who owns it?  
Who approved it?  
What examples does it follow?  
What must it never do?  
What review path applies?  
What happens if the output touches talent, rights, NILP®, claims, or paid media?  
What record should exist before media moves forward?

If those answers are not written down, the company is relying on private judgment instead of company rules.

# **Section 2: What an Operating File Is**

## **Eyebrow**

OPERATING FILE

## **Headline**

An operating file is a company-owned instruction layer.

## **Body**

An operating file defines how a production agent or team should handle a specific production task.

It is more complete than a prompt.

It gives the agent context, limits, examples, review steps, and escalation rules.

## **Operating Files May Include**

Agent role  
Approved tasks  
Blocked actions  
Approved examples  
Rejected examples  
Output formats  
Naming rules  
File handoff rules  
Tone and style notes  
Production checklists  
Review steps  
Escalation notes  
Rights review triggers  
NILP® review triggers  
Owner instructions  
Risk level  
Last review date

## **Supporting Line**

The prompt tells the agent what to do.  
The operating file tells the company how the work is controlled.

# **Section 3: Prompt vs Operating File**

## **Eyebrow**

THE DIFFERENCE

## **Headline**

A prompt gets a result. An operating file creates repeatable practice.

## **Body**

A loose prompt may be useful, but it usually does not carry enough structure for real production.

## **Loose Prompt**

Written by an individual.  
Stored privately.  
Changed without review.  
May not include blocked actions.  
May not include approved examples.  
May not define human approval.  
May not define rights or NILP® triggers.  
May not leave a clear record.

## **Operating File**

Owned by the company.  
Attached to a production agent or production path.  
Includes approved and rejected examples.  
Defines what the agent can and cannot do.  
Defines output format.  
Includes review steps and escalation triggers.  
Connects to human approval gates.  
Can be updated under owner control.

## **Core Line**

A prompt is a tool.  
An operating file is a company asset.

# **Section 4: Why Operating Files Matter**

## **Eyebrow**

WHY IT MATTERS

## **Headline**

Agents cannot follow company rules if the rules are scattered.

## **Body**

Media companies often have strong internal knowledge.

But that knowledge may live in too many places.

Old decks.  
Brand books.  
Producer notes.  
Legal comments.  
Rights spreadsheets.  
Slack threads.  
Talent notes.  
Platform specs.  
Prompt libraries.  
Individual memory.

Production agents cannot reliably follow a company process if the process is scattered across all of those places.

Operating files make the rules visible.

They give agents and teams one clear place to understand the task, examples, limits, review path, and escalation rules.

# **Section 5: What Should Go Into an Operating File**

## **Eyebrow**

FILE STRUCTURE

## **Headline**

The file should explain the task, the limits, and the review path.

## **Body**

Creation Skills designs operating files around the production job.

The goal is not to create a long policy document.

The goal is to create usable instructions.

## **Core Sections**

### **Agent Role**

What the production agent supports and where it fits in the production path.

### **Approved Tasks**

What the agent is allowed to prepare.

### **Blocked Actions**

What the agent must never decide or perform.

### **Approved Examples**

Examples the company wants the agent to follow.

### **Rejected Examples**

Examples the company does not want repeated.

### **Output Format**

The structure, format, naming convention, or delivery style expected.

### **Review Steps**

Who reviews the output before it moves forward.

### **Escalation Triggers**

What causes the work to pause, route, or escalate.

### **Owner Instructions**

Who maintains the file and approves updates.

# **Section 7: Blocked Actions Are Essential**

## **Eyebrow**

BLOCKED ACTIONS

## **Headline**

The file should say what the agent cannot do.

## **Body**

Many companies write prompts that only tell agents what to produce.

That is not enough.

Responsible agent use also requires clear no-go areas.

## **Common Blocked Actions**

Cannot publish final media.  
Cannot approve talent usage.  
Cannot approve NILP® usage.  
Cannot change rights terms.  
Cannot approve product claims.  
Cannot bypass legal escalation.  
Cannot override brand review.  
Cannot reuse restricted assets.  
Cannot approve paid media use.  
Cannot approve sponsor use.  
Cannot approve final delivery.  
Cannot change operating files without owner approval.  
Cannot make legal determinations.

## **Outcome**

The agent can support production without becoming an uncontrolled decision maker.

# **Section 10: Operating Files and Learning**

## **Eyebrow**

LEARNING LOOP

## **Headline**

The file should improve from approved work.

## **Body**

Operating files should not stay frozen.

They should improve from real production.

Approved outputs, rejected drafts, final edits, legal comments, brand corrections, producer notes, client feedback, and delivery issues can all improve the file.

But material changes should be controlled.

## **Learning Rule**

The agent can suggest an update.  
The owner approves it.  
The operating file changes.  
The next output improves.

## **Update Signals**

Approved captions  
Rejected drafts  
Final edits  
Brand corrections  
Legal comments  
Rights review findings  
Producer notes  
Client feedback  
Delivery issues  
Format changes  
Market changes  
Team corrections

## **Owner Controls**

Approve update  
Reject update  
Revise file  
Adjust blocked action  
Update examples  
Change review path  
Raise risk level  
Request additional human review

# **Section 11: Signs You Need an Operating File**

WARNING SIGNS

## **Headline**

The need appears when a prompt starts affecting real work.

## **Body**

A prompt probably needs to become an operating file when it starts supporting repeated production.

## **Warning Signs**

Multiple people use the same prompt.  
The output appears in real assets.  
The agent prepares public-facing copy.  
The prompt touches talent-led media.  
The prompt works from unreleased files.  
The prompt prepares paid media variants.  
The prompt drafts product claims.  
The prompt supports translations.  
The prompt suggests reuse.  
The prompt affects metadata.  
No one owns updates.  
No one knows the blocked actions.  
No one knows what review path applies.

## **Supporting Line**

If a prompt affects real media, it needs company ownership.
